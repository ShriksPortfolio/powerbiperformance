Workspace Roles — Key Considerations Best Practice
Consideration Detail
Licensing Admin, Member, and Contributor roles require Power BI Pro or PPU. Viewers can access content without Pro only if the workspace is in Premium/Fabric capacity.
Orphaned Workspaces A workspace without an Admin is "orphaned". Always ensure at least 2 Admins per workspace. Fabric Admins can recover orphaned workspaces via the Admin Portal.
RLS Behaviour Row-Level Security (RLS) only applies to users with the Viewer role. Admins, Members, and Contributors bypass RLS — they have edit permissions on the semantic model.
Group Assignment For governed content, assign security groups (not individuals) to workspace roles. If a user is in multiple groups, the highest permission level applies.
Principle of Least Privilege Assign the lowest role necessary. Most report consumers should be Viewers. Content creators should be Contributors. Only workspace owners should be Admins.
Maximum Workspaces A Pro user can be a member of a maximum of 1,000 workspaces.

Why Unit Test Semantic Models? — Business Case Foundation Quality
Risk Without Testing What Goes Wrong Business Impact
Silent wrong numbers A DAX measure returns incorrect totals under specific filter combinations. The report looks fine at the top level — it's only wrong when the business drills down or exports to Excel. Executive decisions based on inaccurate figures. Erodes trust in the entire analytics platform. Recovery cost is exponentially higher the later it's caught.
Referential integrity violations Orphaned foreign keys in the fact table (e.g., a SegmentID of -1) produce (Blank) rows in visuals. Users see unexplained blanks in slicers and matrices. Perceived data quality failures. Users revert to Excel workarounds. Adoption stalls because "the numbers don't match".
Model regression after change A developer adds a new dimension, changes a relationship, or modifies a measure. Existing reports silently break — totals shift, filters stop propagating, or (Blank) rows appear. Change becomes feared rather than embraced. Teams avoid model improvements. Technical debt accumulates. CI/CD adoption is blocked.
Export-to-Excel mismatch Totals shown in Power BI visuals don't match the totals when the same table is exported. Caused by context transition issues, implicit measures, or visual-level filters not captured in export. Finance teams lose confidence. Reconciliation becomes a manual monthly exercise. Audit risk in regulated industries (financial services, insurance).
RLS bypass or data leakage A role filter is misconfigured. An Admin/Member/Contributor sees unfiltered data (by design), but a user is accidentally assigned a role that grants broader access than intended. Regulatory breach (GDPR, FCA). Sensitive data exposed to unauthorised users. Potential fines and reputational damage.
Refresh failures in production A schema change in the upstream data source breaks the Power Query refresh. The dataset serves stale data, sometimes for days, before anyone notices. Business operates on outdated data. Scheduled refresh failures cascade into missed SLAs. Operations team firefighting instead of building.

Unit Testing Pattern — Target vs Results Design Pattern
Component Description Implementation
Unit Testing Target A reference table of expected outcomes. Each row defines a test: ID, description, and the expected (baseline) value. This is the "contract" the model must satisfy. Maintained as a standalone table (Excel, SQL, or embedded in the model). Updated whenever the model changes intentionally. Acts as the single source of truth for validation.
Unit Testing Results
(Base vs Target)		 The test execution layer. Runs each test against the current model and compares the result to the target value. Reports PASS/FAIL per test with the actual value returned. Implemented as DAX queries (EVALUATE + ROW + UNION pattern) or Fabric Notebooks. Each test returns: Test ID, Test Type, Test Name, Semantic Model Result, Target, PASS/FAIL flag.
Unit Testing Results
(Detail)		 For failed tests, the detail level shows the actual offending rows — e.g., which specific records have invalid foreign keys, negative values, or dates outside range. A drill-through or filtered query that returns Source, Date, Test ID, and the key columns (UnitID, SegmentID, SalesChannelID, etc.) for each violation, enabling root cause analysis.
When to Run Semantic Model Unit Tests Test Triggers
Trigger What to Run Frequency Gate / Action
After model change (development) All test categories: Structure, Relationships, Data Quality, Measure Accuracy, Totals, Security, Performance, BPA Every commit / pull request Block deployment to Test/UAT if any test fails. Run in Azure Pipelines or Fabric Notebooks as part of CI/CD.
After data refresh Structure (row counts), Relationships (orphaned keys), Data Quality (NULLs, negatives, date range) Every scheduled refresh Alert via Power Automate or Teams webhook if tests fail. Optionally prevent report refresh from completing.
Before promotion (Dev → Test → Prod) Full regression suite: all test categories with production baseline values Each deployment pipeline stage Deployment pipeline gate: must pass before promotion. Integrated with Azure DevOps release approval.
Upstream schema change Structure (schema shape), Relationships (new orphaned keys), Data Quality (new NULLs or unexpected values) Triggered by data engineering team notification or automated schema drift detection Block refresh until model is updated to handle the schema change. Notify model owner.
Periodic governance review BPA rules, Security (RLS coverage), Performance benchmarks, documentation completeness Monthly or quarterly Feed results into Governance Dashboard. Inform CoE review meetings. Track trends over time.

Unit Testing in the Deployment Pipeline (CI/CD) Enterprise Pattern
Stage Pipeline Stage Tests Executed Gate Criteria Outcome on Failure
1 Pull Request (Build) BPA rules (Tabular Editor CLI). Schema validation. Naming convention checks. Zero BPA violations (error level). No implicit measures. No bi-directional relationships. PR blocked. Developer must fix before merge.
2 Dev Workspace (Validate) Deploy to Dev workspace. Run full DAX test suite: Structure, Relationships, Data Quality, Measure Accuracy, Totals. All tests PASS. Row counts within ±1% of target. All measures within ±0.01 of baseline. Deployment to Test blocked. Alert to developer with failing test details.
3 Test/UAT Workspace Full regression suite with production data volumes. Security tests (RLS "View As"). Performance benchmarks (DAX Studio Server Timings). All tests PASS. Page load < 3s. No measure > 500ms. RLS filters correctly for all test users. Promotion to Prod blocked. Findings routed to CoE for review.
4 Production (Post-Deploy) Smoke tests: key measures validated against known baselines. Refresh success. Row count sanity check. Smoke tests PASS within 15 minutes of deployment. Refresh completes within SLA window. Rollback to previous version. Incident raised. Post-incident review scheduled.

Faster, safer changes Regression tests catch unintended side effects of model changes instantly. Developers can refactor with confidence. Reduced time-to-deploy for model changes. Fewer production incidents caused by model regression.

Self-Service Group Management — Key Decisions Governance Decision
Decision Option A: Centralised Option B: Decentralised Recommendation
Who creates groups? IT only. Full control, slower turnaround. CoE leads / champions via self-service. Faster, but requires training. Hybrid: IT creates tenant-level groups; workspace-level groups delegated to trained owners.
Who manages membership? IT manages all group memberships. Designated group owners (e.g., team leads) add/remove members. Decentralised with group owners. Microsoft recommends self-service group management with designated owners.
How are groups audited? Manual quarterly review by IT. Automated via Microsoft Graph API + access reviews in Entra ID Governance. Automated access reviews (Entra ID P2). Flag stale memberships. Quarterly manual review as backup.
Naming convention? Free-form naming by creators. Enforced prefix/suffix naming policy in Entra ID. Enforce a naming policy: SG-PBI-[Scope]-[Role]. Makes groups discoverable and auditable.

Star schema is the optimal structure for VertiPaq. Clean separation of facts and dimensions enables predictable filter propagation and correct DAX totals. HIGH Star Schema Design

Every column has a memory cost in VertiPaq. Removing unused columns reduces model size, improves compression, and speeds up refresh. Remove, don't just hide. HIGH Power Query Best Practices

9 Co-locate gateway with SQL Server Install the gateway on a machine in the same network segment as the SQL Server. Minimise network hops. Avoid WAN links between gateway and database. Network latency between gateway and SQL Server directly impacts refresh duration and DirectQuery response time. Microsoft recommends minimising distance. HIGH Gateway Deployment
10 Schedule refreshes during off-peak database hours Align Power BI refresh schedules with periods of low SQL Server activity. Avoid running refreshes during batch processing or month-end reconciliation windows. Competing workloads on SQL Server degrade both the refresh performance and the transactional system. Scheduling avoids resource contention. HIGH Data Refresh
11 Enable data streaming on the gateway In gateway config, set StreamBeforeRequestCompletes = True in the gateway DLL config file to enable streaming mode. By default, the gateway spools all data before sending to Power BI. Streaming mode sends data as it's retrieved — improving performance for large datasets. MID Gateway Deployment
12 Monitor gateway performance counters Use Windows Performance Monitor to track CPU, memory, and disk I/O on the gateway machine. Set alerts for sustained CPU > 80% or memory > 85%. Proactive monitoring prevents refresh failures. A gateway running at 90%+ CPU will queue refreshes, causing timeouts and cascading failures. MID Gateway Sizing
13 Avoid merging queries from different SQL databases If merging data from two SQL databases, do the join in SQL (cross-database query or linked server) rather than merging two Power Query tables. Merging across different data sources in Power Query breaks query folding entirely. Both tables are loaded into memory and joined locally by the mashup engine. MID Query Folding Basics
14 Use query reduction settings for DirectQuery In Power BI Desktop: File → Options → Query reduction. Enable "Add an Apply button to each slicer" and "Add an Apply button to all basic filters". Without query reduction, every slicer change fires a SQL query through the gateway. With Apply buttons, users build their filter context and fire one query. MID Dataset Modes
15 Limit visuals per page for DirectQuery reports Keep visuals to 8 maximum per page. Each visual generates a separate SQL query. Use bookmarks and drill-through instead of dense pages. 10 visuals = 10 simultaneous SQL queries per page load. For DirectQuery, this directly impacts SQL Server and gateway load. Fewer visuals = faster page render. MID Gateway Sizing
16 Create columnstore indexes for DirectQuery On large fact tables queried via DirectQuery, create non-clustered columnstore indexes (NCCI) to accelerate analytical queries. Columnstore indexes are optimised for the aggregate/filter patterns Power BI generates. Can improve DirectQuery performance by 10-100x on large tables. MID Gateway Deployment
17 Use Power BI aggregation tables for dual mode Create summary tables in Import mode paired with detail tables in DirectQuery. Configure aggregation mappings in Power BI Desktop. Aggregations give Import-level performance for common queries while falling back to DirectQuery for detail-level drill-through. Best of both worlds. MID Aggregations
18 Test refresh duration and set timeout thresholds Measure baseline refresh times. Set pipeline alerts if refresh exceeds 2x baseline. Default gateway timeout is 2 hours — adjust if needed. Refresh duration creep is a leading indicator of data growth or query degradation. Without monitoring, timeouts appear as sudden failures rather than gradual degradation. MID Data Refresh

Outdated gateways lose support, miss performance improvements, and may have known bugs. Unsupported versions are the first thing Microsoft checks in support tickets. LOW Gateway Installation

Q1 What does the Power BI Administrator actually own — and what do they not own? ▶

They own the platform configuration: tenant settings, capacity allocation, gateways, admin roles, sensitivity-label enablement, and tenant-wide monitoring.

They do not own data content, semantic models, the business meaning of data, or the RLS rules inside workspaces. The mantra is "govern the platform, not the content."

Q2 Where are the Admin's guardrails — what must they never do unilaterally? ▶
  • Enable a risky tenant feature (Publish to Web, export) for the entire org without scoping to a security group and going through change control.
  • Grant the Fabric Administrator role broadly — it breaks least privilege.
  • Define the sensitivity-label taxonomy alone — that's Security / Compliance.
  • Change capacity SKU or budget without Finance and the CoE.
  • Edit or republish content inside business workspaces.
Q3 Which decisions are the Admin's, and which escalate to the governance board / CoE? ▶

Admin decides: day-to-day platform configuration within agreed policy — scoping a setting to a group, assigning a workspace to a capacity, rotating a gateway.

Escalates: new policy itself (e.g. turning Publish to Web on at all), label taxonomy, capacity budget, and any exception to data-residency or compliance rules.

Q4 How is the Power BI Admin different from a Capacity Admin and a Workspace Admin? ▶
  • Tenant / Fabric Admin — the whole tenant: tenant settings, every capacity, all admin roles.
  • Capacity Admin — one specific capacity: assign workspaces and manage workloads, but no tenant settings.
  • Workspace Admin — a single workspace's content and members only.

By least privilege, most people need workspace or capacity admin; only a tiny number need tenant admin.

Q5 What's the least-privilege way to grant the Fabric Administrator role? ▶
  • Keep the number of tenant admins tiny — typically 2–4.
  • Use Entra PIM for just-in-time, time-boxed elevation rather than standing access.
  • Assign via a security group, not named individuals; audit membership quarterly.
  • Prefer scoped roles (Capacity Admin, workspace roles) wherever the task allows.
Q6 How should the Admin manage tenant settings safely? ▶

Document a baseline of every setting and its intended state. Scope each setting to a security group, treat changes as change-controlled, and review the exported tenant-settings configuration on a regular cadence. Use delegated admin settings so domain / workspace admins handle scoped concerns without needing tenant rights.

Q7 What is the Admin accountable for monitoring? ▶

The activity / audit log (via the Admin API or Purview), the admin monitoring workspace and usage metrics, capacity utilisation and throttling, gateway health, and feature usage that drifts from policy (e.g. a sudden spike in exports). The Admin surfaces the signal; the CoE and Security act on the breach.

Q1 How does a Capacity Admin differ from a Workspace Admin — and both from the tenant Fabric Admin? ▶
  • Tenant / Fabric Admin — the whole tenant: tenant settings and every capacity.
  • Capacity Admin — one capacity: assign workspaces, manage workloads, scale, and configure capacity settings.

Q2 What can a capacity contributor do versus a capacity admin? ▶

A capacity admin can change capacity settings, add contributors, and add or remove workspaces. A contributor can assign workspaces to the capacity but can't modify settings or delete it.

Contributor is the safe delegation: it lets workspace admins move their workspace onto a managed capacity without handing them full control.

Q3 How do you know a capacity is overloaded, and what are the options? ▶

Watch the Capacity Metrics app. Fabric first smooths usage, then throttles in stages as sustained demand grows:

  • Interactive delay — 20s delay once you exceed the next 10 minutes of CUs.
  • Interactive rejection — interactive requests rejected (60-minute window).
  • Background rejection — everything rejected (24-hour window).

When it's sustained: optimise the content, scale up the SKU, scale out by moving workspaces, and turn on surge protection. Optimise before you buy.

Q4 What's the governance thinking behind workspace placement and multiple capacities? ▶

Isolation and a small blast radius. Put exec / critical content on its own capacity; keep dev-test off production; give each business unit its own capacity so management can be decentralised and chargeback is clean. If one capacity is mismanaged, the impact is confined to it — not the whole org.

Q5 How should workspace roles be assigned safely? ▶
  • Assign to security groups, not named individuals.
  • Least privilege: most consumers = Viewer (which also enforces RLS on browse), authors = Contributor, owners = Member.
  • Reserve Admin for the few who actually manage access — don't sprinkle it.
Q6 Where are this role's guardrails — what escalates upward? ▶

They run the day-to-day within delegated authority. What escalates: buying a new capacity or changing budget / SKU strategy (tenant Admin + Finance), tenant settings, and anything touching tenant-wide policy. They surface the signals — throttling, orphaned workspaces, runaway items — and the tenant Admin / CoE own the policy response.

Q1 What does the Security Lead own versus the Power BI Admin? ▶

The Security Lead owns the what and why: the label taxonomy, RLS / OLS standards, DLP policy, conditional-access requirements, and classification policy.

The Power BI Admin owns the enablement: turning the labels / DLP capability on in the tenant and scoping the settings. Three hands, one chain — Security defines the rule, the Admin flips the switch, the developer implements it in the model.

Q2 RLS vs OLS — when do you use each, and what's the #1 pitfall? ▶
  • RLS filters rows (a salesperson sees only their region) via DAX role filters.
  • OLS hides tables / columns — and even their existence and metadata — via model roles (set with Tabular Editor).

Both apply only to Viewers; Admin / Member / Contributor bypass them. The #1 pitfall: roles are additive — a user in two roles sees the union, so "once denied" is never guaranteed. Prefer one role per user, and always Test-as-role.

Q3 Static vs dynamic RLS — what should the standard be? ▶

Static = a fixed filter per role ([Region]="West"). Dynamic = one role that filters by the signed-in user with USERPRINCIPALNAME() against a user-mapping table — far more scalable.

Standard: prefer dynamic RLS, design deny-by-default (an unknown user returns no rows), and never rely on a fall-through TRUE() that leaks everything to an unexpected value.

Q4 How does DLP for Power BI work, and what are its limits? ▶

DLP policies (Microsoft Purview) scan semantic models in Premium / Fabric workspaces on publish, republish, and refresh — matching sensitive-info types or sensitivity labels — then raise policy tips and admin alerts.

Limits: it targets semantic models (not every item), it's not real-time (logs lag), and it skips service-principal-owned models. For real-time blocking, use Defender for Cloud Apps.

Q5 Where does the Security Lead's authority stop? ▶

They don't enable the tenant capability (that's the Admin), don't author RLS in every model (developers do), and don't build content. They set standards, review, and gate. The aim is to make the secure path the standard path — not to be the bottleneck every model queues behind.

Q6 How do you keep data protected once it leaves Power BI? ▶
  • Sensitivity labels with encryption travel with exports — Excel, PDF, PowerPoint, .pbix — and Analyze-in-Excel.
  • Default + mandatory label policies catch unlabeled content.
  • Export tenant settings restrict risky formats; Defender for Cloud Apps can block downloads of Highly-Restricted content in real time.

Protection has to follow the data — not stop at the service boundary.

Q1 What does the Data Architect own — and is it a technical or a strategic role? ▶

Both, deliberately. They own the blueprint: the reference architecture, the modelling standards, and the single-source-of-truth strategy.

But a large part of the job is non-technical — translating business strategy into an architecture, running design sessions, setting standards, and aligning stakeholders. Q2 Why is the star schema the standard — in one breath? ▶

Because every report visual fires a query that filters, groups, and summarises. A star schema gives you dimensions to filter / group and facts to summarise — exactly how the engine wants to work. It's faster, smaller, and far easier for authors to understand than a normalised, operational layout.

The usual smells: snowflaking, mixing fact and dimension in one table, and many-to-many relationships.

Q3 How does the architecture flow from raw source to a report? ▶

The reference pattern:

source → bronze (raw) → silver (cleansed, conformed) → gold (star-schema) → certified semantic model → reports

Medallion layers raise quality stage by stage; the gold layer is shaped as a star schema; the semantic model is the single, governed, reusable layer the business actually builds on.

Q4 One enterprise semantic model, or many? Where's the line? ▶

Neither extreme. One giant model for everything is impractical; a new model per report is chaos. The architect's call is to lean toward few, shared, certified semantic models that deliver a single version of the truth — and create a new one only when a genuinely distinct subject area justifies it.

Reuse via live connection and Build permission beats duplication every time.

Q5 How do you set standards without becoming the bottleneck? ▶

Make the standard the easy path: reference-architecture docs, template models, a clear certification checklist, and mentoring — so teams self-serve correctly. The architect reviews and gates the major models, not every report. Governance by enablement, not by approval queue.

Q6 Where does the Architect's authority stop? ▶

They design the pattern and own the standards, but they don't run the platform (Admin / Capacity Admin), don't build the pipelines (Data Engineer), and don't author every model (BI Developer). They're the design authority and the alignment layer — accountable for coherence, not for every line of M or DAX.

Q2What is query folding, and why obsess over it?▶

Folding pushes your Power Query steps back to the source as a single native query, so the source does the work and only filtered / aggregated results travel. Folded queries are dramatically faster and lighter.

Break folding and the engine pulls everything, then transforms in memory — slow and costly. Put Remove Columns and filters early so they fold.

Q3When and why incremental refresh?▶

For large fact tables, refreshing everything nightly is wasteful and fragile. Incremental refresh partitions by date and refreshes only the recent window while keeping history — faster refreshes, less capacity, fewer failures.

Pair it with query folding: the source must be able to fold the RangeStart / RangeEnd filters.

Q4Dataflows Gen2 vs pipelines vs notebooks — when each?▶
  • Dataflows Gen2 — low-code Power Query prep; great for reusable, business-friendly transformations.
  • Pipelines (Data Factory) — orchestration: move data, sequence activities, schedule.
  • Notebooks (Spark) — code-first, big-data transformation and complex logic.

Often combined: a pipeline orchestrates notebooks / dataflows across the medallion layers.

Q5How do you make refreshes genuinely reliable?▶

Treat it as an operations discipline, not a one-time setup: sequence dependencies, add retries and failure alerts, monitor run history, reconcile row counts to source, and catch source schema drift early. The business only trusts the dashboard if it's fresh — reliability is a feature.

Q6Where does the Engineer's responsibility stop?▶

At the gold-layer / semantic-model boundary. They don't define business meaning (governance / SMEs), build the DAX model or reports (Developer), own the architecture (Architect), or manage capacity (Capacity Admin). They own data arriving correct and on time.

Q2What are the top DAX performance pitfalls?▶
  • Iterating (row-by-row) where you could aggregate.
  • Calculated columns that should be measures.
  • Bidirectional filters and high-cardinality columns.
  • Not using variables to avoid recomputation.

Q3How do you keep a model performant?▶
